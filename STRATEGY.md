## 1. 사이트 정체 (Identity)

**Cellab은 B2B/B2G 고관여 상담 funnel형 사이트입니다. 쇼핑몰이 아닙니다.**

- 정량펌프는 고가·기술 제품 — 사용자가 "이 모델 살게요" 하고 즉시 결제하는 시나리오 없음
- 사용자는 본인이 뭐가 필요한지 모르는 상태로 진입 → 가이드·위저드에 의존
- 마지막 전환은 견적 문의 → 사장님 직접 응대 → 나비엠알오·올포랩 송출 또는 직거래
- 가격은 견적제. 사이트에 가격 강조 X

### 정체성 확인 — GA4 근거 (2026-05-15 스냅샷, 30일)

| 페이지 | 조회수 | 인당 조회 | 평균 체류 | 해석 |
|---|---|---|---|---|
| 정량펌프 응용 셋업 가이드 | 178 (32%) | 10.47 | 1분 52초 | 진입 1번지 |
| 정량펌프, 어떤 걸 골라야 하나요 (recommend2 위저드) | 110 (20%) | 27.5 | **7분 55초** | 진지한 의사결정 도구 |

## 2. 마스터 동선 (Master Funnel)

```
[유입 채널]
  구글·네이버 검색 → 셀렙 블로그 (학술·시리즈 글)
  구글 직접 → 메인 (위저드·셋업 가이드 노출)

       ↓

[의사결정 도구]
  1) 정량펌프 응용 셋업 가이드 (응용별 케이스)
  2) 30초 위저드 (recommend2)
  3) 펌프셋업 리뷰 (실제 사례)
  4) Leadfluid 카탈로그 (deep link 목적지)

       ↓

[전환]
  견적 문의 → 사장님 직접 응대 → 나비엠알오·올포랩 송출 또는 직거래
       ↓
  알고리즘 가속 (재구매·검색 노출 향상)
```

이 동선에 거스르는 작업·결정은 금지. 메모리 `feedback_cellab_master_funnel.md` 위배.

## 4. 메인(index.html) 구성 원칙

다음 순서를 권장:

1. **히어로**: "어떤 펌프가 필요하세요? 30초 위저드" → recommend2 진입 강조
2. **응용 카드 4~6개**: 응용별 셋업 가이드 deep link
3. **펌프셋업 리뷰 카드 3~4개**: 실제 사례
4. **블로그 최신글 2~3개**: 권위·SEO
5. **견적 CTA**: 항상 보이는 버튼
6. **카탈로그 메뉴 1칸**: 직접 진입자용 보조

## 5. 하지 말 것 (Don'ts)

**쇼핑몰형 패턴 금지:**

- 가격(₩) 표시 강조 — 견적제 신뢰 깨짐
- 장바구니 / 즉시 결제 UI
- 회원 등급·포인트
- 재고 표시
- 리뷰 별점·평점 (B2B에서는 사례 글이 더 강함)
- 단순 상품 나열형 페이지 (pump.html이 그 흔적이었고, 2026-05-15에 삭제·leadfluid로 redirect 완료)
- "BEST · NEW · SALE" 같은 마케팅 태그

**위저드·가이드 약화 금지:**

- 위저드 단계 줄이려는 단순화 (현재 7분 55초 체류가 본질)
- 가이드 콘텐츠 축약 (학술 톤 유지가 신뢰 자산)
- 블로그 발행 빈도 저하

## 6. admin DB 관리 방향 (SSOT)

### 6.0 절대 원칙 — admin DB가 모든 데이터의 기준 (2026-05-18 사장님 통보)

**홈페이지의 모든 작업·결정은 admin에서 편집 가능한 데이터를 기준으로 한다.**

사장님이 admin에 들어가 한 값을 바꾸면, 그 변경이 사이트 전체에 자동 반영되어야 한다. 이걸 만족시키지 못하는 작업은 "사장님이 관리·제작 어려워진다"고 명시적으로 알리고 사장님 승인 후 진행.

#### SSOT에 포함되어야 할 것
- 펌프 카탈로그 (catalog.json) — 펌프 사양·매칭·헤드·이미지·그룹·URL
- 블로그 글 메타 (posts.json) · 셋업 리뷰 (setups.json + parts.json)
- 그룹 정의·라벨·매칭 어휘·유량 경계값 등 메타 설정 (catalog.json 또는 settings.json)
- 위저드 질문·답변 옵션 어휘 (settings.json 또는 catalog.json)

#### 금지
- HTML/JS의 변수·상수에 운영 가능한 데이터를 박지 말 것
- "사장님이 운영 중 바꿀 수 있어야 하는 값"은 무조건 admin에서 편집 가능

#### AI 에이전트 검증 체크리스트 (작업 전 매번)
- [ ] 이 작업에 새 데이터를 코드에 박을 가능성이 있는가? 있으면 사장님 알림 후 진행
- [ ] 기존 데이터를 코드에서 끌어 쓰는가? 그 데이터는 admin에서 편집 가능한가?
- [ ] 데이터 구조 변경 시 admin UI도 같이 갱신했는가?
- 위 셋 중 하나라도 NO면 사장님께 "관리·제작 어렵다"고 알릴 의무

자세한 안전 규칙은 `CRITICAL_RULES.md § 7` 참조.

### 통합 우선순위

| 우선 | 데이터 | 현 위치 | 통합 방향 |
|---|---|---|---|
| 1 | **펌프 63종 사양·매칭·이미지** | ✅ `_build/catalog.json` 단일 SSOT (Phase 1~3 완료, 2026-05-15). admin_pumps.json 폐기, leadfluid.html 하드코딩 제거 | (완료) |
| 2 | **펌프셋업 데이터 (펌프+튜브+헤드 조합)** | `_build/setups.json` + `_build/parts.json` | 그대로 유지, admin에 편집 UI 추가 |
| 3 | **블로그 글 메타** | `_build/posts.json` | admin에 발행 메타 편집 추가 |
| 4 | **응용별 가이드 콘텐츠** | 가이드 페이지 직접 편집 | 신규 — 응용별 페이지 자동 생성 가능하게 |
| 5 | **마켓 URL (allforlab·navimro)** | `_build/products.json` | catalog.json 흡수 |
| 6 | **이미지 자산** | `/img/leadfluid/` 등 | GitHub Contents API로 admin 업로드 |

### admin.html 재설계 (탭 구조)

```
┌─────────────────────────────────────────┐
│ Cellab 관리자                            │
├─────────────────────────────────────────┤
│ [펌프 카탈로그] [셋업 리뷰] [블로그]      │
│  [응용 가이드] [통계]                    │
└─────────────────────────────────────────┘
```

**탭 1: 펌프 카탈로그 (catalog.json)** ← 최우선 ROI
- 86개 펌프, 한 펌프당 30+ 필드 (id, name, series, flow, ip, comm, applications, matching{flow/func/ctrl/fluid/channels}, heads, image, allforlab/navimro URL)
- 행별 펼침형 편집 (기본 보기엔 핵심 컬럼만 → 클릭하면 풀 편집)
- 이미지 업로드 기능
- 위저드 매칭 키워드 직접 편집 ← **위저드 7분 55초 체류가 견적 전환으로 이어지려면 추천 정밀도 결정. 가장 큰 임팩트**

**탭 2: 셋업 리뷰 (setups.json)**
- 펌프 + 튜브 + 헤드 조합 카드
- 사진·사용처·인용 글

**탭 3: 블로그 글 메타 (posts.json)**
- slug, 제목, 메타, 발행일, 네이버 cellab01 URL, 키워드
- (본문 HTML은 별도 파일 — 본문 편집은 IDE에서)

**탭 4: 응용 가이드 (신규)**
- 응용별 페이지를 카드 데이터로 관리
- 데이터 → 정적 페이지 자동 빌드

**탭 5: 통계 (선택)**
- 위저드 응답 빈도 (어떤 조합이 자주 들어오는지)
- 견적 문의 누적
- inquiry로 들어온 wizSummary 누적

### admin UI에 가격·재고 강제 입력 만들지 말 것

견적제이므로 가격 필드는 선택. 마켓 URL만 있으면 송출 가능.

## 8. 의사결정 체크리스트

새 작업·기능 검토 시 다음을 점검:

- [ ] 이 작업이 마스터 동선 어느 단계에 있는가? (검색·진입 → 의사결정 → 전환 → 직거래·송출)
- [ ] 위저드·가이드·리뷰·블로그·견적 funnel을 강화하는가, 약화하는가?
- [ ] 쇼핑몰형 패턴(가격강조·장바구니·평점 등)을 도입하지 않는가?
- [ ] **admin SSOT 절대 원칙** (§ 6.0)을 거스르지 않는가? — 코드에 데이터 박는 형태면 NO
- [ ] 단계가 너무 크지 않은가? (CRITICAL_RULES — 한 번에 큰 변경 금지)
- [ ] GA4·네이버 통계·인콰이리 추세에 기반한 결정인가? (감 아님)

이 중 하나라도 NO면 진행 전 사장님과 재논의.
